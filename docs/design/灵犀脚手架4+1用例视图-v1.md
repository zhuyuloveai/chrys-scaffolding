# 灵犀脚手架 4+1 用例视图 v1（基线版）

## 1. 文档目的

本文作为灵犀脚手架方案的 `+1` 视图，即用例视图。

当前版本作为基线版，目标是先固定主业务链路、系统边界和主要参与者，为后续逻辑视图提供输入。

目标只有一个：

> 先用业务语言讲清楚“谁在用系统、系统帮他完成什么、系统边界在哪里”。

本文不讨论数据库、服务拆分、技术框架和部署细节，这些留给后续逻辑视图和软件实现设计文档。

## 2. 系统范围

灵犀脚手架是一个独立的企业配置服务，负责：

- 用户登录与企业鉴权
- 识别用户身份、部门、角色和权限范围
- 向用户展示允许范围内的配置项和管理能力
- 生成统一配置结果 `ResolvedScaffoldConfig`
- 通过 `ChrysAdapter` 将统一配置转换成 `chrys` 可消费的配置结果 `ChrysLaunchBundle`
- 触发 `chrys` 启动

`chrys` 不是脚手架的一部分，而是脚手架的消费端。

补充边界：

- 在部分企业集成场景下，脚手架还依赖 `chrys` 提供的受控运行时扩展能力
- 其中一类是运行时动态注入 / provider 能力，用于凭证按需获取、续期和注入
- 另一类是会话结束回调能力，用于审计、打点和上报

## 3. 当前已明确的业务前提

以下内容已在当前讨论中收敛，可作为后续用例分析的前提。

### 3.1 登录后用户上下文最小模型

登录完成后，系统至少拿到四项基础信息：

- `employee_id`
- `token`
- `department`
- `user_type`

其中：

- `employee_id` 用于标识当前用户
- `token` 用于访问企业市场和企业服务
- `department` 用于确定部门范围
- `user_type` 用于确定用户所属角色类型

补充约束：

- 这四项组成当前阶段最小的 `LoginContext`
- `token` 是登录后拿到的访问凭证，不是用户稳定主数据
- token 如何持久化属于实现层问题，当前不在用例视图展开

### 3.2 用户类型

当前阶段固定三种用户类型：

- `user`
- `department_admin`
- `super_admin`

### 3.3 权限范围的产生方式

权限范围不是用户身份字段本身，而是由脚手架后端根据以下两项统一解析：

- `department`
- `user_type`

也就是说：

- 用户身份回答“这个人是谁”
- 权限范围回答“这个人能看什么、能选什么、能管什么”

### 3.4 角色边界的基本定义

- `user`：只负责使用系统
- `department_admin`：负责管理本部门范围内的能力与策略
- `super_admin`：负责管理全局范围内的能力与策略

### 3.5 hook / 动态注入能力的角色定位

当前阶段认为：

- 会话结束后的审计、打点、上报更适合作为受控生命周期回调能力
- 凭证准备和运行中的续期更适合作为动态注入 / provider 能力
- 这两类能力都属于系统侧或管理侧能力
- 它们都不属于普通用户需要直接关注和配置的核心能力项

## 4. 参与者

### 4.1 主要参与者

- 普通用户
- 部门管理员
- 超级管理员

### 4.2 外部系统参与者

- 企业统一鉴权系统
- 企业组织/权限服务
- 企业市场或模板服务
- `chrys`

### 4.3 内部自动参与者

- 脚手架配置装配器
- `ChrysAdapter`
- 生命周期执行器

## 5. 参与者目标

### 普通用户

- 登录系统
- 获取适合自己的默认配置
- 在允许范围内选择模型、MCP、Skill 等能力
- 生成并启动自己的 `chrys`

补充说明：

- `agent` 当前先沿用 `chrys` 默认能力，不作为脚手架 v1 的重点个性化配置项

### 部门管理员

- 完成普通用户的所有操作
- 进入部门级管理页面
- 管理本部门可用的能力项和范围策略

### 超级管理员

- 完成全局管理操作
- 管理公共能力市场、能力源和全局策略

### 企业统一鉴权系统

- 完成登录认证
- 返回用户工号和可调用企业服务的 `token` 或等价会话凭证

### 企业组织/权限服务

- 根据用户工号返回部门、角色和权限上下文

### chrys

- 消费脚手架交付的适配后配置
- 按配置启动运行时

### ChrysAdapter

- 将统一配置映射为 `chrys` 当前可消费的原生配置
- 控制脚手架与 `chrys` 之间的配置适配边界

## 6. 角色边界对照表

| 用户类型 | 作用域 | 主要页面 | 主要操作 | 不应具备的能力 |
|---|---|---|---|---|
| `user` | 个人使用范围 | 登录页、配置选择页、启动确认页 | 登录鉴权、查看默认配置、在允许范围内选择模型/MCP/Skill、生成统一配置、启动 `chrys` | 不管理白名单、不管理部门策略、不管理公共能力项、不直接配置 hook |
| `department_admin` | 本部门范围 | 普通用户页面 + 部门管理页 | 拥有 `user` 全部能力；管理本部门用户权限范围；管理本部门能力项来源、可见范围、可选范围和默认策略 | 不应管理全局公共策略；不应管理其他部门配置 |
| `super_admin` | 全局范围 | 普通用户页面 + 部门管理页 + 全局管理页 | 拥有 `department_admin` 全部能力；管理全局公共能力市场、全局能力项和全局策略；拥有所有配置权限 | 无部门范围限制 |

## 9. 核心用例说明

## UC-01 用户登录并完成鉴权

### 目标

让用户获得一个有效登录态，并拿到后续访问企业服务所需的身份信息。

### 前置条件

- 用户尚未登录，或本地登录态已失效

### 主流程

1. 用户打开脚手架
2. 脚手架引导用户完成统一鉴权流程
3. 鉴权系统返回用户工号和 token
4. 脚手架形成当前用户的 `LoginContext`

### 结果

系统至少获得一份可用于后续企业 API 调用的 `LoginContext`

### 备注

本用例只解决“用户是谁”和“是否已登录”，不直接决定具体权限。

## UC-02 用户获取默认配置与可选范围

### 目标

让用户在登录后看到与自己身份匹配的默认配置和可选配置。

### 前置条件

- 用户已完成鉴权

### 主流程

1. 脚手架根据用户工号查询组织与权限上下文
2. 脚手架得到用户的部门、角色和权限范围
3. 系统根据权限范围裁剪公共配置与模板市场
4. 系统生成该用户的默认配置
5. 系统向用户展示默认配置和可选范围

### 结果

用户能看到：

- 默认配置
- 允许选择的模型、MCP、Skill
- 是否显示管理入口

普通用户视角下，hook 不作为直接配置重点向用户暴露。

补充说明：

- `agent` 当前沿用 `chrys` 默认能力，不在 v1 中作为单独配置项展示

### 关键约束

用户看到的不是全量配置，而是权限过滤后的结果。

## UC-03 用户调整个性化配置

### 目标

让用户在允许范围内做个性化选择。

### 前置条件

- 用户已经拿到默认配置与可选范围

### 主流程

1. 用户浏览当前默认配置
2. 用户在允许范围内修改自己的选择
3. 系统实时校验用户选择是否越权
4. 系统保存当前选择

### 结果

形成一份待装配的用户选择结果。

### 关键约束

- 用户只能选被授权的配置项
- 用户不能直接修改超出授权范围的公共策略

## UC-04 系统生成统一配置

### 目标

把公共配置、能力目录结果、用户选择和企业约束装配成一份统一配置结果。

### 前置条件

- 用户已完成个性化选择

### 主流程

1. 系统读取用户当前选择
2. 系统合并默认配置、能力目录结果和用户选择
3. 系统注入企业约束、凭证引用和系统侧运行要求
4. 系统生成统一配置结果 `ResolvedScaffoldConfig`

### 结果

输出一份统一配置结果 `ResolvedScaffoldConfig`

### 关键约束

这一步产出的是脚手架内部统一结果，不直接等价于 `chrys` 当前原生配置。

## UC-05 系统适配并交付 chrys 可用配置

### 目标

把统一配置结果转换成 `chrys` 当前可消费的原生配置集合。

### 前置条件

- 系统已成功生成 `ResolvedScaffoldConfig`

### 主流程

1. 系统读取 `ResolvedScaffoldConfig`
2. `ChrysAdapter` 将模型相关配置映射为 `model profile` 和 `.env updates`
3. `ChrysAdapter` 将 Skill、MCP 等配置映射为 `agent profile`，其中 Agent 骨架暂沿用 `chrys` 默认能力
4. 系统生成启动所需参数和交付结果 `ChrysLaunchBundle`
5. 系统将适配结果写入目标位置或准备交付给 `chrys`

### 结果

输出一份 `ChrysLaunchBundle`

当前阶段建议至少包含：

- `model_profile_yaml`
- `env_updates`
- `agent_profile_yaml`
- `launch_args`

### 关键约束

- 适配层只负责把统一配置映射到 `chrys` 当前支持的能力边界
- 不应把 `chrys` 当前尚未稳定支持的能力当作已可直接消费的正式配置面

## UC-06 用户启动 chrys

### 目标

让用户基于脚手架生成的配置启动自己的 `chrys`。

### 前置条件

- 系统已成功生成 `ChrysLaunchBundle`

### 主流程

1. 用户确认启动
2. 脚手架基于 `ChrysLaunchBundle` 拉起 `chrys`
3. `chrys` 加载其原生配置并启动运行时
4. 用户进入 `chrys` 运行环境

### 结果

用户获得与自身身份匹配的个性化 `chrys`

## UC-07 管理员进入管理页面

### 目标

让管理员获得与其管理身份匹配的管理入口。

### 前置条件

- 用户已登录
- 用户具备管理员角色

### 主流程

1. 系统识别用户管理角色
2. 系统向用户展示对应管理入口
3. 用户进入管理页面

### 结果

用户获得附加的管理能力，而不是只看到普通用户页面。

## UC-08 管理员管理部门级能力项与范围策略

### 目标

让部门管理员管理本部门范围内的能力项和策略。

### 前置条件

- 用户为部门管理员

### 主流程

1. 管理员进入部门管理页面
2. 查看本部门当前能力项来源、可见范围、可选范围和默认策略
3. 新增、修改或下线部门级能力项与部门策略
4. 系统校验其是否超出部门管理范围
5. 系统保存变更

### 结果

部门级能力项与策略被更新，并影响该部门用户后续看到的默认配置或可选范围。

## UC-09 超级管理员管理公共能力市场与全局策略

### 目标

让超级管理员管理全局公共配置能力。

### 前置条件

- 用户具备全局管理权限

### 主流程

1. 超级管理员进入全局管理页面
2. 管理公共能力源、市场条目、全局可见策略和全局默认策略
3. 系统保存变更

### 结果

公共配置与全局策略被更新，并影响后续所有相关用户。

## UC-10 系统在启动前准备企业凭证

### 目标

确保 `chrys` 启动时具备访问企业平台所需的有效凭证。

### 前置条件

- 用户已准备启动 `chrys`
- 统一配置或适配结果中声明了企业凭证依赖

### 主流程

1. 系统检查启动所需凭证
2. 系统初始化或刷新相关凭证
3. 系统将凭证注入最终启动上下文

### 结果

`chrys` 启动时具备访问企业服务的必要凭证。

### 备注

本用例主要由脚手架侧启动编排与凭证提供机制完成，不依赖通用 hook。

## UC-11 系统在会话结束后执行审计或上报

### 目标

在 `chrys` 会话结束后完成必要的企业后置动作。

### 前置条件

- `chrys` 会话已结束
- 统一配置或适配结果中声明了会话结束动作

### 主流程

1. `chrys` 通过受控会话结束回调通知系统
2. 系统读取会话结束时需要执行的动作
3. 系统执行审计、打点或上报流程

### 结果

企业要求的后置流程被执行完成。

## 12. 统一能力市场的补充用例抽象

### 12.1 为什么要在用例层统一表达

- 长期看 `agent / skill / mcp` 在运行时语义不同
- 但在脚手架控制面里，它们都表现为“可被展示、选择、授权、管理的能力项”
- 因此在用例视图里，先统一表达为“能力市场”更合适

当前 v1 先聚焦 `skill / mcp`，`agent` 配置保留为后续专题。

后续进入逻辑视图时，再分别展开三类能力项的装配差异。

### 12.2 当前建议的两层拆分

在脚手架里，需要把“来源”和“管理”分开理解：

- 来源：这项能力从哪里来
- 管理：这项能力谁能看、谁能选、谁能管、谁默认拿到

这能避免把“企业市场同步”“个人导入”“部门共享”混成一件事。

### 12.3 当前建议的统一对象

- `CapabilityItem`：最终展示给用户或管理员的能力项
- `CapabilitySource`：能力项的来源记录
- `CapabilityPolicy`：能力项的可见、可选、可管理、默认启用策略

在用例层先关心这三个对象即可，不必提前落到数据库表结构。

### 12.4 当前已收敛的枚举

- `source_type`
  - `enterprise_sync`
  - `personal_import`
  - `batch_import`
- `owner_scope`
  - `personal`
  - `department`
  - `global`

补充说明：

- `source_type` 表示能力项是怎么进入脚手架的
- `owner_scope` 表示能力项归谁持有和谁对它负责
- `owner_scope` 不等于最终的可见范围或可选范围

### 12.5 v1 推荐支持矩阵

| 能力类型 | v1 推荐支持的来源组合 | 当前建议 |
|---|---|---|
| `mcp` | `enterprise_sync + global`、`personal_import + personal`、`batch_import + department`、`batch_import + global` | `mcp` 最适合先做“导入 + 共享 + 范围控制”；v1 先不做脚手架内的通用配置编辑器 |
| `skill` | `enterprise_sync + global`、`batch_import + department`、`batch_import + global` | `skill` 更偏运营分发对象；v1 先不开放个人上传进入市场 |
| `agent` | 暂不纳入 v1 主线能力市场 | 当前统一沿用 `chrys` 默认 Agent 能力；后续如确有必要，再单独展开 Agent 模板与覆盖策略 |
