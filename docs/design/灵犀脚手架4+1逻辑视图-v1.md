# 灵犀脚手架 4+1 逻辑视图 v1（基线版）

## 1. 文档目的

本文作为灵犀脚手架方案的逻辑视图。

当前版本作为基线版，目标是先固定：

- 系统内部的核心对象
- 各对象之间的职责边界
- 从登录、裁剪、装配到交付的主逻辑链路
- 脚手架与 `chrys` 之间的适配边界

本文不讨论：

- 数据库表设计
- 服务拆分与部署拓扑
- API 字段细节
- 物理落盘路径和进程启动细节

这些内容留给后续软件实现设计文档。

## 2. 输入前提

本视图基于用例视图中的以下结论：

- 登录后用户上下文最小模型固定为 `employee_id / token / department / user_type`
- 用户类型固定为 `user / department_admin / super_admin`
- 权限范围由 `department + user_type` 统一解析
- `model` 需要按用户群体和权限范围裁剪可见与可选集合
- 当前 v1 优先将 `skill / mcp` 作为控制面能力项管理，`agent` 配置先沿用 `chrys` 默认能力
- 脚手架先生成统一配置 `ResolvedScaffoldConfig`
- 再通过 `ChrysAdapter` 转换为 `chrys` 可消费结果 `ChrysLaunchBundle`

补充约束：

- `token` 属于登录后上下文的一部分
- token 的安全持久化策略属于实现层问题，当前不在逻辑视图展开

## 3. 逻辑视图总原则

### 3.1 控制面和运行面分离

- 组织、权限、来源、范围策略属于脚手架控制面
- `chrys` 只消费适配后的运行结果

### 3.2 统一配置先于运行时适配

- 脚手架内部先形成统一配置
- `chrys` 当前配置形态只在适配层出现

### 3.3 权限先裁剪，再做默认装配

- 不允许先生成默认配置，再做权限剔除
- 默认值必须在授权后的候选集合中产生

### 3.4 适配层只映射当前已支持能力

- `ChrysAdapter` 只负责把统一配置映射到 `chrys` 当前稳定支持的配置面
- 对 `chrys` 尚未稳定支持的能力，不应在适配层强行落地

## 4. 分层结构

逻辑上建议拆为四层：

1. 身份与访问层
2. 候选目录层
3. 配置装配层
4. 适配交付层

逻辑结构如下：

```text
[Identity / Access]
    |
    +--> LoginContext
    +--> UserAccessContext
    |
    v
[Candidate Catalog]
    |
    +--> ModelOption
    +--> ModelPolicy
    +--> ModelCatalog
    +--> CapabilitySource
    +--> CapabilityItem
    +--> CapabilityPolicy
    +--> CapabilityCatalog
    |
    v
[Config Resolution]
    |
    +--> DefaultPolicy
    +--> UserSelection
    +--> ConfigAssembler
    +--> ResolvedScaffoldConfig
    |
    v
[Delivery / Adapter]
    |
    +--> ChrysAdapter
    +--> ChrysLaunchBundle
    |
    v
[chrys]
```

## 5. 核心对象与职责

### 5.1 身份与访问对象

| 对象 | 作用 | 说明 |
|---|---|---|
| `LoginContext` | 表示登录后可用上下文 | 包含 `employee_id / token / department / user_type`，用于后续访问企业 API 与权限解析 |
| `UserAccessContext` | 表示用户当前访问上下文 | 是权限解析后的结果，决定可见、可选、可管范围 |
| `AccessContextResolver` | 负责访问上下文解析 | 根据 `department + user_type` 生成访问上下文 |

### 5.2 模型候选对象

| 对象 | 作用 | 说明 |
|---|---|---|
| `ModelOption` | 表示一个可供选择的 model 候选项 | 受用户群体和权限范围控制，最终由适配层映射为 `chrys` 的 model 生效配置 |
| `ModelPolicy` | 表示 model 的范围策略 | 描述 model 的可见、可选、默认和可管理范围 |
| `ModelCatalog` | 表示裁剪后的 model 候选集合 | 面向当前用户的 model 可选集，当前阶段建议单次启动只选择一个当前生效 model |

补充说明：

- `model` 在控制面里也需要权限裁剪
- 但当前阶段不强行并入 `CapabilityItem`
- 它更适合作为独立的候选配置对象参与装配

### 5.3 能力目录对象

| 对象 | 作用 | 说明 |
|---|---|---|
| `CapabilitySource` | 表示能力项来源 | 关注能力从哪里进入脚手架 |
| `CapabilityItem` | 表示对用户可展示的能力项 | 当前阶段主要承载 `skill / mcp` 的市场、选择和管理对象 |
| `CapabilityPolicy` | 表示能力项策略 | 描述可见、可选、可管理、默认启用范围 |
| `CapabilityCatalog` | 表示裁剪后的能力目录 | 面向当前用户或管理员的候选能力集合 |

补充说明：

- `agent` 当前不纳入 v1 主线能力目录
- 后续如果确有必要控制面化，再单独补 `agent` 专题设计

### 5.4 配置装配对象

| 对象 | 作用 | 说明 |
|---|---|---|
| `DefaultPolicy` | 表示默认装配规则 | 决定各槽位默认值如何产生 |
| `UserSelection` | 表示用户个性化选择 | 只记录允许覆盖的选择结果 |
| `ConfigAssembler` | 负责统一配置装配 | 合并默认、model 候选、能力项、用户选择和企业约束 |
| `ResolvedScaffoldConfig` | 表示脚手架统一配置结果 | 是适配层输入，不直接暴露为 `chrys` 原生配置 |

### 5.5 适配交付对象

| 对象 | 作用 | 说明 |
|---|---|---|
| `ChrysAdapter` | 负责适配 `chrys` | 将统一配置转换为 `chrys` 原生配置集合 |
| `ChrysLaunchBundle` | 表示交付给 `chrys` 的结果 | 至少包含 `model_profile`、`env_updates`、`agent_profile`、`launch_args` |
| `LaunchOrchestrator` | 负责启动交付 | 根据适配结果落盘、注入或拉起 `chrys` |

### 5.6 生命周期对象

| 对象 | 作用 | 说明 |
|---|---|---|
| `CredentialRequirement` | 表示凭证依赖 | 描述哪些运行目标依赖哪些凭证 |
| `LifecycleAction` | 表示声明式生命周期动作 | 如启动前 `ensure_credential`、会话结束 `audit_report` |
| `LifecyclePlan` | 表示生命周期动作集合 | 属于统一配置的一部分，其中会话结束类动作需要映射到 `chrys` 的会话结束回调能力 |

## 6. 核心对象关系

### 6.1 身份与访问关系

```text
LoginContext
    -> AccessContextResolver
    -> UserAccessContext
```

含义：

- `LoginContext` 表示登录后当前用户可用的最小上下文
- `UserAccessContext` 表示“这个人当前能做什么”

### 6.2 模型候选关系

```text
ModelOption
    -> ModelPolicy
    -> ModelCatalog
```

含义：

- `ModelOption` 回答“当前有哪些 model 候选项”
- `ModelPolicy` 回答“哪些人能看到、能选、默认拿到哪个 model”
- `ModelCatalog` 是当前用户真正可见的 model 候选集合

### 6.3 能力目录关系

```text
CapabilitySource
    -> CapabilityItem
    -> CapabilityPolicy
    -> CapabilityCatalog
```

含义：

- `CapabilitySource` 回答“能力从哪里来”
- `CapabilityItem` 回答“当前系统里有什么能力项”
- `CapabilityPolicy` 回答“谁能看、谁能选、谁能管、谁默认拿到”

### 6.4 装配关系

```text
UserAccessContext
+ ModelCatalog
+ CapabilityCatalog
+ DefaultPolicy
+ UserSelection
    -> ConfigAssembler
    -> ResolvedScaffoldConfig
```

含义：

- 装配器不直接理解组织规则的原始来源
- 它只消费已经解析好的访问上下文、model 候选集合和能力目录
- `model` 与 `skill / mcp` 在装配阶段共同进入统一配置，但运行时落点不同
- `agent` 当前不作为脚手架单独装配对象，先按 `chrys` 默认能力处理

### 6.5 适配关系

```text
ResolvedScaffoldConfig
    -> ChrysAdapter
    -> ChrysLaunchBundle
    -> chrys
```

含义：

- `ResolvedScaffoldConfig` 是脚手架内部真相
- `ChrysLaunchBundle` 是 `chrys` 侧可消费结果

## 7. 关键子模型

### 7.1 LoginContext

建议在逻辑上至少包含：

- `employee_id`
- `token`
- `department`
- `user_type`

这里的重点不是字段多少，而是固定这个对象的语义：

- 它不是稳定的用户主数据
- 它是登录完成后系统手里拿到的最小可用上下文
- 后续访问企业 API、解析权限、裁剪能力目录都以它为起点

### 7.2 UserAccessContext

建议在逻辑上至少包含：

- `employee_id`
- `department`
- `user_type`
- `visibility_scope`
- `selection_scope`
- `management_scope`
- `default_scope`

这里的重点不是字段多少，而是固定这个对象的语义：

- 身份和权限上下文已经完成解析
- 后续模块不再直接回头处理原始组织规则

### 7.3 ModelOption

建议在逻辑上至少包含：

- `id`
- `name`
- `provider`
- `model_id`
- `endpoint_ref`
- `owner_scope`
- `status`
- `profile_template_ref`

其中：

- `provider / model_id` 描述最终要接入的模型
- `endpoint_ref` 或等价引用指向模型接入端点信息
- `profile_template_ref` 用于支持后续映射到 `chrys` 的 model profile 结构

补充约束：

- 凭证不直接作为 `ModelOption` 的固有字段
- 凭证仍更适合放在 `integration_intent` 一侧管理

### 7.4 ModelPolicy

建议逻辑上至少区分：

- `visibility_scope`
- `selectable_scope`
- `manageable_scope`
- `default_enable_scope`

补充说明：

- `model` 也需要同样的范围策略语义
- 但当前阶段不要求与 `CapabilityPolicy` 强行合并成同一个对象

### 7.5 CapabilityItem

建议在逻辑上至少包含：

- `id`
- `type`
- `name`
- `description`
- `source_id`
- `owner_scope`
- `status`
- `payload_ref`

其中：

- `type` 当前主要指 `skill / mcp`
- `owner_scope` 只表示归属，不等于最终可见范围
- `payload_ref` 表示具体配置负载的引用，不要求在逻辑视图里展开为物理路径

### 7.6 CapabilitySource

当前已收敛三类来源：

- `enterprise_sync`
- `personal_import`
- `batch_import`

这个对象负责回答：

- 能力项怎么进来的
- 谁对它的来源接入负责

但它不直接回答：

- 谁能看到它
- 谁能选择它

这些属于 `CapabilityPolicy`

### 7.7 CapabilityPolicy

建议逻辑上至少区分：

- `visibility_scope`
- `selectable_scope`
- `manageable_scope`
- `default_enable_scope`

这些策略是控制面的核心，不应直接下发给 `chrys`。

### 7.8 ResolvedScaffoldConfig

这是脚手架内部最重要的装配结果。

建议逻辑上分成三块：

- `runtime_intent`
- `integration_intent`
- `lifecycle_intent`

其中：

- `runtime_intent` 描述最终要启用的模型、MCP、Skill
- `integration_intent` 描述企业集成与凭证依赖
- `lifecycle_intent` 描述启动前后需要执行的声明式动作

可以更直白地理解为：

- `runtime_intent` 回答“这次启动最终跑什么”
- `integration_intent` 回答“这些运行时能力要接到企业系统，还依赖哪些接入条件”
- `lifecycle_intent` 回答“在启动前后，系统还要补做哪些动作”

补充约束：

- 这里仍然是脚手架统一模型
- 不要求一比一等于 `chrys` 当前 schema

补充说明：

- 当前阶段建议 `runtime_intent.model` 只表达一个当前生效 model
- 也就是说，单次启动先按“一个主 model”处理
- `agent` 当前不作为独立装配槽位进入 `runtime_intent`，统一沿用 `chrys` 默认 Agent 能力

当前阶段建议先把三块内容边界固定为：

- `runtime_intent`
  - 当前主线只关心最终生效的 `model / mcp / skill`
  - 这里保存的是最终启用结果，而不是候选目录、权限范围或市场原始数据
- `integration_intent`
  - 当前主线只关心运行这些能力所依赖的企业接入条件
  - 典型内容包括凭证依赖、服务端点引用、请求头或环境变量引用
  - 这里应保留依赖关系，不直接保存敏感 token 明文
  - 运行中的凭证续期与按需取值主要依赖 `chrys` 侧动态注入 / provider 能力
- `lifecycle_intent`
  - 当前主线只关心围绕运行时前后发生的声明式动作
  - v1 重点先收敛启动前凭证准备、会话结束审计/上报这两类动作
  - 其中启动前凭证准备由脚手架启动编排与凭证提供机制完成
  - 会话结束审计/上报依赖 `chrys` 的会话结束回调能力

### 7.9 ChrysLaunchBundle

这是适配层输出给 `chrys` 的原生结果集合。

当前阶段建议至少包含：

- `model_profile_yaml`
- `env_updates`
- `agent_profile_yaml`
- `launch_args`

补充说明：

- `model` 在当前 `chrys` 里主要通过 `model profile + .env` 生效
- `skill / mcp` 主要通过 `agent profile` 生效
- 当前阶段产出的 `agent_profile_yaml` 以 `chrys` 默认 Agent 骨架为基底，脚手架主要补充 `skill / mcp` 相关结果
- 因此这一步不是“传一份总配置文件”，而是“产出一组原生配置结果”

## 8. 主逻辑链路

### 8.1 登录与访问解析

```text
统一鉴权
-> LoginContext
-> AccessContextResolver
-> UserAccessContext
```

这一步产出的核心结果是：

- 登录后当前用户可用上下文是什么
- 当前用户的访问范围是什么

### 8.2 候选目录裁剪

```text
ModelOption / ModelPolicy
-> 按 UserAccessContext 裁剪
-> ModelCatalog

CapabilitySource / CapabilityItem / CapabilityPolicy
-> 按 UserAccessContext 裁剪
-> CapabilityCatalog
```

这一步产出的核心结果是：

- 当前用户的 `ModelCatalog`
- 当前用户的 `CapabilityCatalog`

### 8.3 默认装配与用户选择

```text
ModelCatalog
+ CapabilityCatalog
+ DefaultPolicy
+ UserSelection
-> ConfigAssembler
-> ResolvedScaffoldConfig
```

这一步产出的核心结果是：

- 系统默认选哪个 model
- 系统默认怎么选
- 用户最后改成什么
- 企业约束最终如何落入统一配置

#### 8.3.1 装配输入基线

当前阶段建议 `ConfigAssembler` 至少消费以下输入：

- `UserAccessContext`
- `ModelCatalog`
- `CapabilityCatalog`
- `DefaultPolicy`
- `UserSelection`

补充约束：

- 候选集合必须先完成权限裁剪
- 装配阶段不再回头处理原始组织规则
- 不允许把未授权候选项直接带入最终配置

#### 8.3.2 装配顺序基线

当前阶段建议固定以下顺序：

1. 以 `ModelCatalog / CapabilityCatalog` 作为全部可装配候选集合
2. 根据 `DefaultPolicy` 为各槽位计算系统默认结果
3. 在允许覆盖的范围内叠加 `UserSelection`
4. 注入企业固定约束、凭证依赖、生命周期动作等系统结果
5. 生成最终 `ResolvedScaffoldConfig`

补充说明：

- 第 2 步产出的默认值必须来自已授权候选集合
- 第 3 步只允许覆盖可选、可覆盖的槽位
- 第 4 步不再改变权限边界，只补充系统侧必须存在的运行要求

#### 8.3.3 各槽位装配规则基线

当前阶段建议先固定三类槽位：

- `model`
  - 单次启动只允许一个当前生效 model
  - 默认值必须从 `ModelCatalog` 中产生
  - 如果用户显式选择了一个仍然可选的 model，则用户选择优先
  - 如果用户没有选择，或原选择已失效，则回退到默认策略重新计算
- `mcp`
  - 作为多选能力项参与装配
  - 系统先根据 `DefaultPolicy` 得出默认启用集合
  - 用户可在允许范围内增删自己的启用结果
- `skill`
  - 作为多选能力项参与装配
  - 系统先根据 `DefaultPolicy` 得出默认启用集合
  - 用户可在允许范围内增删自己的启用结果

补充说明：

- `agent` 当前不作为独立装配槽位，固定沿用 `chrys` 默认 Agent 能力
- `integration_intent` 与 `lifecycle_intent` 在运行时槽位结果确定后再补齐

#### 8.3.4 优先级与冲突处理基线

当前阶段建议固定以下优先级：

1. 企业固定约束
2. 用户在授权范围内的显式选择
3. 部门级默认策略
4. 全局默认策略
5. 系统兜底策略

其中：

- 企业固定约束优先级最高，可强制启用、强制禁用或强制指定
- 用户选择只有在“仍然授权且允许覆盖”时才能生效
- 用户历史选择如果已不再授权，应在装配时自动失效并回退默认结果
- “可见但不可选”的能力项，可以展示给用户，但最终状态由系统策略决定

补充约束：

- 对必需槽位，如果授权候选集合为空，装配应返回显式错误，而不是静默生成不完整配置
- 对可选槽位，如果默认结果和用户结果都为空，可以合法为空

### 8.4 适配与启动

```text
ResolvedScaffoldConfig
-> ChrysAdapter
-> ChrysLaunchBundle
-> LaunchOrchestrator
-> chrys
```

这一步产出的核心结果是：

- `chrys` 当前真实可消费的配置集合
- 实际启动参数和交付方式

## 9. ChrysAdapter 的边界

`ChrysAdapter` 不是业务装配器，而是运行时适配器。

它只负责两件事：

1. 把统一配置翻译成 `chrys` 当前原生配置
2. 屏蔽 `chrys` 当前配置形态对脚手架内部模型的污染

当前阶段建议明确以下边界：

- `department`
- `user_type`
- 权限策略
- 能力来源
- 市场可见规则

这些都不应该下发给 `chrys`。

当前阶段适合映射给 `chrys` 的内容主要是：

- 生效中的模型配置
- 生效中的 MCP 配置
- 生效中的 Skill 配置
- 默认 Agent 骨架的引用或派生结果
- 企业凭证引用和少量环境变量
- 声明式生命周期动作的可支持子集

补充说明：

- `ChrysAdapter` 不只负责静态配置映射
- 对于选中的 model，需要按 `chrys` 当前 `model profile + .env` 机制完成映射
- 当前 v1 不在脚手架侧定义独立的 Agent 模板和覆盖规则，而是默认复用 `chrys` 自带 Agent 能力
- 对于 `integration_intent` 中的凭证依赖与注入目标，还需要映射到 `chrys` 当前可提供的动态注入 / provider 能力
- 启动前 `ensure_credential` 由脚手架侧启动编排先完成，不依赖通用 hook
- 对于 `lifecycle_intent` 中的会话结束类动作，还需要映射到 `chrys` 当前提供的会话结束回调能力
- 因此脚手架与 `chrys` 的边界不仅包括配置适配，也包括动态注入 / provider 适配和会话结束回调适配

## 10. 当前约束

### 10.1 对 chrys 现状的约束

当前讨论默认基于以下现实：

- `chrys` 当前主配置入口不是一份统一大配置
- `model` 主要通过 `model profile + .env` 生效
- `skill / mcp` 主要通过 `agent profile` 生效
- `agent` 骨架当前沿用 `chrys` 默认能力，不在 v1 中做独立控制面建模

因此逻辑视图里保留统一配置，但适配层输出必须面向 `chrys` 当前真实能力。

### 10.2 生命周期回调与动态注入能力的约束

当前阶段建议明确区分两类能力：

- 会话结束回调能力
- 运行时动态注入 / provider 能力

其中：

- 会话结束后的审计、打点、上报依赖前者
- 凭证准备、运行中续期、headers/env 动态取值依赖后者

而不是统一抽象为脚手架向 `chrys` 下发任意脚本 hook。
