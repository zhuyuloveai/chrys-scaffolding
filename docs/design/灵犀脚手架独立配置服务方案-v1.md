# 灵犀脚手架独立配置服务方案 v1

## 1. 背景

本文基于以下两部分上下文整理：

- 原始草图整理稿：[pic/README.md](./pic/README.md)
- 当前对 `chrys` 的能力梳理与讨论结论

原稿里“灵犀脚手架”一开始容易被理解成 `chrys` 内部的一组配置页面，但经过补充澄清后，需求应重新定义为：

> 灵犀脚手架是一个独立的配置服务。它可以复用与 `chrys` 类似的 TUI 交互架构，但在系统边界上与 `chrys` 解耦。它的职责是完成企业身份识别、权限裁剪、公共配置装配和个性化选择，并输出一份可被 `chrys` 消费的个性化启动配置。

这一定义决定了后续设计重点不再是“如何把脚手架嵌入 `chrys`”，而是“如何把脚手架做成控制面，并与 `chrys` 通过稳定协议对接”。

## 2. 核心结论

本方案先固定三个最重要的判断：

1. 灵犀脚手架是独立服务，不是 `chrys` 的一个内置配置页。
2. `chrys` 是运行时消费端，不承载企业权限、组织架构、市场运营等控制面语义。
3. 双方的耦合点应当是“启动配置协议 + 动态注入/provider 协议 + 会话结束回调协议”，而不是内部代码结构。

换句话说：

- 灵犀脚手架负责“决定用户应该拿到什么配置”
- `chrys` 负责“拿到配置后如何运行”

## 3. 系统定位

### 3.1 灵犀脚手架的定位

灵犀脚手架是企业内的配置控制面，负责：

- 登录与企业鉴权
- 判定用户所属部门、角色、权限范围
- 管理公共配置和模板市场
- 结合权限与用户选择生成最终配置
- 将最终配置交付给 `chrys`

### 3.2 chrys 的定位

`chrys` 是本地或半本地的 agent 运行时，负责：

- 消费由脚手架生成的最终配置
- 启动模型、Agent、MCP、Skill 等运行时能力
- 提供少量与企业平台相关的运行时动态注入 / provider 能力
- 提供受控的会话结束回调能力

### 3.3 两者的关系

两者不是父子模块关系，而是控制面与运行面的关系：

```text
[灵犀脚手架 / 控制面]
    |
    | 生成 RenderedChrysConfig
    v
[chrys / 运行面]
    |
    | 执行 runtime + dynamic injection/provider + session-end callback
    v
[模型 / MCP / Skill / 企业平台]
```

## 4. 目标与非目标

### 4.1 目标

本期方案的目标是建立一条清晰、可演进的主链路：

- 用户登录并完成企业鉴权
- 系统按部门和角色裁剪可用配置范围
- 用户在允许范围内选择个性化配置
- 系统生成最终的 `chrys` 启动配置
- `chrys` 基于该配置启动，并按约定执行必要的企业集成动作

### 4.2 非目标

以下内容不作为当前阶段必须完成的目标：

- 直接改造 `chrys` 成为企业配置中心
- 在 `chrys` 内部实现完整的组织、权限、市场运营逻辑
- 第一版就支持复杂的任意脚本 hook
- 第一版就支持完整的多端动态热更新
- 第一版就建设完整的运营后台和审批流

## 5. 设计原则

### 5.1 控制面与运行面分离

组织、权限、白名单、运营市场等都属于控制面语义，不应泄漏到 `chrys` 的内部运行时模型中。

### 5.2 输出物化后的最终配置

脚手架输出给 `chrys` 的应当是“最终运行配置”，而不是需要 `chrys` 二次理解的权限规则或模板继承关系。

### 5.3 尽量避免把企业逻辑塞进 chrys

如果 `chrys` 需要理解“部门 A 能看到模型集 X，部门 B 只能看到模型集 Y”，说明边界已经切错。正确做法应是脚手架先完成裁剪，再把裁剪后的结果下发给 `chrys`。

### 5.4 优先稳定协议，再谈 UI 细节

如果启动配置协议、会话结束回调协议、凭证/provider 协议没有先定，后续无论是 TUI 还是服务 API 都会反复返工。

## 6. 原始需求的重新表达

根据原稿，灵犀脚手架要解决的问题可以重新表述为：

> 系统提供一个带权限控制的企业配置入口。用户登录并完成企业鉴权后，系统按部门、角色和权限范围裁剪其可见配置；用户可在允许范围内选择模型、MCP、Skill 和企业集成能力；系统将公共配置、本地偏好与企业约束装配为最终的 `chrys` 启动配置，并在 `chrys` 运行时通过动态注入/provider 机制支持凭证准备与续期，并通过会话结束回调支持审计上报等企业活动。`agent` 配置当前先沿用 `chrys` 默认能力，保留为后续专题。

## 7. 总体架构

### 7.1 分层结构

建议整体拆为四层：

1. 身份与权限层
2. 配置与市场层
3. 配置装配层
4. 交付与启动层

### 7.2 架构示意

```text
[登录 / TUI]
    |
    v
[W3 / 企业鉴权]
    |
    v
[身份与权限层]
    |
    +--> 用户
    +--> 部门
    +--> 角色
    +--> 白名单 / 可见范围 / 可选范围
    |
    v
[配置与市场层]
    |
    +--> 模型模板
    +--> Agent 模板（后续专题）
    +--> MCP 模板
    +--> Skill 包
    +--> Hook 模板
    +--> 企业集成模板
    |
    v
[配置装配层]
    |
    +--> 公共配置
    +--> 个性化选择
    +--> 本地偏好
    +--> 企业约束
    |
    v
[RenderedChrysConfig]
    |
    v
[chrys]
    |
    +--> runtime
    +--> lifecycle
    +--> credential provider
```

## 8. 核心领域模型

建议灵犀脚手架内部至少显式建模以下对象。

### 8.1 身份与权限对象

- `UserIdentity`
- `Department`
- `Role`
- `PermissionPolicy`
- `VisibilityScope`
- `SelectionPolicy`

这些对象属于脚手架内部控制面，不应直接下发给 `chrys`。

### 8.2 配置与模板对象

- `ModelTemplate`
- `AgentTemplate`
- `MCPTemplate`
- `SkillPackage`
- `HookTemplate`
- `IntegrationTemplate`

补充说明：

- 当前 v1 优先做实 `ModelTemplate / MCPTemplate / SkillPackage / IntegrationTemplate`
- `AgentTemplate` 先保留为扩展对象，当前运行时统一沿用 `chrys` 默认 Agent 能力

这些对象用于运营、分发和选择，也主要存在于脚手架控制面。

### 8.3 装配输出对象

- `RenderedChrysConfig`

这是脚手架交给 `chrys` 的最终产物，也是最重要的对接协议对象。

## 9. 与 chrys 的边界定义

### 9.1 脚手架负责什么

- 企业登录与鉴权
- 用户、部门、角色识别
- 权限裁剪
- 配置模板和市场管理
- 个性化选择
- 生成最终配置
- 将最终配置安全交付给 `chrys`

### 9.2 chrys 负责什么

- 解析并消费最终配置
- 启动模型、Agent、MCP、Skill
- 维护运行时上下文与会话
- 执行有限的生命周期扩展逻辑
- 管理运行时凭证刷新

### 9.3 不应由 chrys 承担的职责

- 组织权限判定
- 市场可见性规则
- 部门白名单逻辑
- 模板继承与运营分发逻辑
- 企业控制台级别的管理语义

## 10. 输出给 chrys 的配置协议

### 10.1 协议目标

脚手架输出给 `chrys` 的不是“规则”，而是“结果”。

也就是说，`chrys` 不需要知道：

- 当前用户属于哪个部门
- 这个配置是哪个市场条目派生来的
- 为什么某些配置被隐藏

`chrys` 只需要知道：

- 当前要启动什么
- 允许启用什么
- 在运行时需要执行哪些生命周期动作

### 10.2 推荐结构

建议最终配置按三块组织：

- `runtime`
- `integration`
- `lifecycle`

#### runtime

负责描述 `chrys` 运行时的核心配置：

- model
- default_agent
- mcp
- skills
- env

补充说明：

- 这里的 `default_agent` 表示当前阶段默认沿用 `chrys` 自带 Agent 骨架

#### integration

负责描述企业平台接入与凭证相关信息：

- credential providers
- enterprise endpoints
- tenant / account context
- refresh policies

#### lifecycle

负责描述运行过程中的生命周期动作：

- before_launch
- after_exit

补充说明：

- 运行中的凭证续期不建议放进通用 lifecycle action
- 这类能力更适合作为动态注入 / provider 机制处理

### 10.3 示例结构

以下为示意性结构，不代表最终字段冻结：

```yaml
version: 1

runtime:
  model:
    provider: openai
    model_id: gpt-5.4
    base_url: https://example.internal/v1
    max_context_tokens: 200000

  agent:
    profile: enterprise-code
    instructions: |
      ...
    tools:
      builtins:
        - filesystem.read
        - filesystem.write
        - search
        - shell
    sub_agents: []

  mcp:
    - name: corp-docs
      transport: http
      url: https://mcp.internal.example/api
      headers_ref: corp_mcp_headers

  skills:
    paths:
      - /Users/example/.chrys/skills
    packages:
      - enterprise-guidelines
      - release-playbook

  env:
    NO_PROXY: localhost,127.0.0.1,.internal.example

integration:
  credentials:
    - provider_id: corp_gateway_token
      type: bearer
      inject_targets:
        - runtime.model
      refresh_before_expiry_seconds: 300

    - provider_id: corp_mcp_token
      type: bearer
      inject_targets:
        - runtime.mcp[corp-docs]
      refresh_before_expiry_seconds: 300

  refs:
    corp_mcp_headers:
      Authorization: Bearer ${credential:corp_mcp_token}

lifecycle:
  before_launch:
    - action: ensure_credential
      provider_id: corp_gateway_token
    - action: ensure_credential
      provider_id: corp_mcp_token

  after_exit:
    - action: audit_upload
      target: https://audit.internal.example/session
      blocking: false
```

## 11. 会话结束回调与凭证机制

### 11.1 不建议第一版直接开放任意脚本 hook

虽然原稿里有“hook 自动触发”这一层，但如果一开始就允许脚手架向 `chrys` 下发任意 shell 命令，会立刻带来以下问题：

- 安全边界不清晰
- 执行失败难以观测
- 难以审计
- 难以限制副作用

因此，第一版更适合采用“声明式 action”而不是“任意脚本”。

### 11.2 建议优先支持的声明式动作

- `ensure_credential`
- `audit_upload`

### 11.3 建议的生命周期动作字段

每个生命周期动作建议至少具备：

- `action`
- `blocking`
- `timeout_seconds`
- `retry_policy`
- `failure_policy`
- `input_ref` 或必要参数

### 11.4 凭证刷新更适合做成 provider 机制

凭证刷新不只是一个普通 hook，更接近运行时依赖管理问题。因此建议设计独立的 `CredentialProvider` 机制，而不是把所有刷新逻辑分散在 hook 里。

建议的最小字段包括：

- `provider_id`
- `credential_type`
- `inject_targets`
- `refresh_before_expiry_seconds`
- `refresh_method`
- `failure_policy`

这样好处是：

- token 生命周期可以被统一管理
- 哪个 token 作用于哪个运行时目标是可追踪的
- refresh 失败时可以统一决定是否阻塞启动

### 11.5 当前建议的依赖边界

当前阶段建议明确区分：

- 会话结束后的审计、打点、上报依赖 `chrys` 的会话结束回调能力
- 启动前凭证准备由脚手架启动编排与 `CredentialProvider` 完成
- 运行中的 token 续期、headers/env 动态取值依赖运行时动态注入 / provider 能力

也就是说：

- 只有 after-exit 类动作显式依赖 `chrys` 的生命周期回调
- 凭证相关能力不建议统一塞进通用 hook

## 12. 市场能力的重新定义

原稿中的 `Agent 市场 / MCP 市场 / Skill 市场` 不应直接等价于 `chrys` 的运行时对象。

在脚手架里，市场更合理的定义是“可运营、可分发、可授权的模板目录”，而不是运行时 primitive。

当前阶段建议优先把 `mcp_template` 和 `skill_package` 做实，`agent_template` 先不进入 v1 主线。

### 12.1 市场对象建议

建议统一抽象为 `CatalogItem`，再按类型区分：

- `agent_template`
- `mcp_template`
- `skill_package`
- `hook_pack`
- `integration_pack`

补充说明：

- 当前 v1 先把 `mcp_template` 和 `skill_package` 做实
- `agent_template` 暂保留为后续扩展类型

### 12.2 CatalogItem 建议字段

- `id`
- `type`
- `name`
- `version`
- `owner_scope`
- `visibility_policy`
- `selection_policy`
- `default_config`
- `override_policy`

### 12.3 设计收益

这样做之后：

- 市场是控制面能力
- 最终运行时仍然输出为统一的 `RenderedChrysConfig`
- 避免“市场对象”和“运行时对象”一一耦合

## 13. 配置来源与装配规则

### 13.1 建议的配置来源

建议至少支持三类来源：

- 公共配置
- 用户个性化选择
- 本地偏好或本地缓存

### 13.2 建议显式定义的权限维度

建议不要只说“有权限 / 没权限”，而是至少区分：

- `visible`：是否可见
- `selectable`：是否可选择
- `editable`：是否可编辑
- `overridable`：是否允许本地覆盖

### 13.3 装配顺序建议

推荐的装配顺序如下：

1. 加载公共模板
2. 根据用户身份与权限裁剪可见范围
3. 根据用户选择生成候选运行配置
4. 合并允许范围内的本地偏好
5. 注入企业约束和凭证引用
6. 输出最终的 `RenderedChrysConfig`

## 14. 交付方式建议

脚手架服务输出配置给 `chrys` 的方式可以有多种，但建议第一版尽量简单。

### 14.1 可选方式

- 直接落成本地配置文件
- 写入本地缓存目录后由 `chrys` 启动
- 通过命令行参数传入配置路径
- 通过本地 IPC/HTTP 传递配置

### 14.2 MVP 推荐

建议 MVP 采用：

- 由脚手架服务在本地生成一份确定性的配置文件
- 以配置路径的方式拉起 `chrys`

原因：

- 最容易调试
- 最容易复现问题
- 与 `chrys` 的耦合最小
- 不需要第一版就引入长连接协议

## 15. MVP 范围建议

### 15.1 建议纳入 MVP 的能力

- 登录与企业鉴权
- 用户、部门、角色识别
- 公共模型配置选择
- 公共 MCP 配置选择
- 公共 Skill 包选择
- 生成最终 `chrys` 配置
- 启动前的凭证初始化
- 会话结束时的审计上报

### 15.2 建议暂缓的能力

- 任意脚本 hook
- Agent 模板市场与字段级覆盖
- 复杂市场运营后台
- 动态热更新配置
- 细粒度字段级审批
- 多级模板继承链
- 运行中持续双向同步控制面状态

## 16. 分阶段推进建议

### 阶段一：协议与配置生成

目标：

- 固定 `RenderedChrysConfig`
- 固定 lifecycle action 定义
- 固定 credential provider 定义

产出：

- 配置 schema
- 示例配置
- 本地文件交付方式

### 阶段二：脚手架服务 MVP

目标：

- 打通登录、权限裁剪、配置选择、最终配置生成

产出：

- 第一版独立 TUI
- 第一版控制面后端
- 第一版本地启动联调链路

### 阶段三：企业集成增强

目标：

- 支持受控的企业平台接入能力
- 增强凭证刷新与审计链路

产出：

- credential manager
- lifecycle runner
- 可观测性与失败处理

## 17. 待确认问题

以下问题建议作为下一轮讨论重点：

1. 输出给 `chrys` 的最终配置格式是否采用单一 YAML 文件。
2. `chrys` 侧是否接受“配置文件路径启动”的交付方式作为第一版协议。
3. 凭证刷新是否独立建模为 `CredentialProvider`，而不是直接塞进 hook。
4. 市场对象是否按模板目录建模，而不是直接暴露运行时对象。
5. 用户是否允许在脚手架生成配置后，在本地继续改动部分字段。
6. 本地偏好是否需要回写到脚手架服务，还是只保留在本地缓存。
7. `chrys` 需要新增哪些最小动态注入 / provider 能力和会话结束回调能力，才能满足企业接入需求。
8. 凭证注入目标是否需要标准化为字段路径或命名引用。

> 灵犀脚手架负责控制面，`chrys` 负责运行面；控制面输出最终配置，运行面消费最终配置并执行有限、受控的生命周期扩展。
