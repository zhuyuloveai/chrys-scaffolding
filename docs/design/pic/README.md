# `pic` 草图整理说明

本文根据 [1.png](./1.png) 到 [6.png](./6.png) 六张草图整理而成，目标是把图片里的想法转成可读的 Markdown 说明。

这几张图存在明显的重复和演进关系，可以统一理解为一套“按部门和权限控制的前端配置/市场接入方案”：

- 用户先登录，并经过 `W3` 鉴权。
- 系统根据用户所属部门决定可见范围和可操作范围。
- 管理侧负责部门白名单、用户权限范围、部门配置管理。
- 业务配置主要包括：模型配置、数据打点 hook、市场、MCP 市场、Skill 市场。
- 配置既有“本地已有配置”，也有“公共配置”，最终会组合后交给前端使用。
- 配置和权限信息会落到 `NoSQL` 数据库及配置文件中。

## 统一后的整体理解

可以把六张图归并成下面这条主流程：

```text
[登录 / 前端 UI]
        |
        v
[W3 鉴权]
        |
        v
[判定用户部门]
        |
        +--> [部门 admin]
        +--> [部门白名单 / 配置权限]
        +--> [用户权限范围管理]
        |
        v
[配置装配层]
        |
        +--> [模型配置]
        +--> [数据打点 hook]
        +--> [市场]
        +--> [MCP 市场]
        +--> [Skill 市场]
        |
        +--> [本地已有配置]
        +--> [公共配置]
        |
        v
[用户按权限选择配置]
        |
        v
[服务层]
        +--> [NoSQL 数据库]
        +--> [配置文件]
```

## 图 2 说明

原图：[2.png](./2.png)

图 2 把“登录鉴权”和“配置选择”两件事拆开表达，说明权限和配置是两个并行但会汇合的过程。

### 关键信息

- 登录流程：
  - 登录
  - `W3` 鉴权
  - 判定用户部门
  - 进入部门管理相关能力
- 部门管理相关能力包括：
  - 部门 admin
  - 对应部门白名单配置权限
  - 用户权限范围管理
- 配置流程：
  - 模型配置先根据用户权限过滤
  - 形成“用户可选择的配置列表”
  - 用户自行选择
- hook 流程：
  - 数据打点 hook 会触发数据上报脚本
  - `conversation` 结束后也可以触发 hook 上传
- 市场能力会进入配置环节。

### 文本框图

```text
[登录] -> [W3 鉴权] -> [判定用户部门]
                               |
                               +--> [部门 admin]
                               +--> [对应部门白名单配置权限]
                               +--> [用户权限范围管理]

[模型配置] -> [按用户权限过滤配置列表] -> [用户自行选择]

[数据打点 hook] -> [通过 hook 触发数据上报脚本]
[conversation 结束] -> [触发 hook 上传]

[市场] -> [配置]
```

## 图 3 说明

原图：[3.png](./3.png)

图 3 更偏后端视角，描述前端能力如何进入服务层，以及服务层如何管理权限和持久化配置。

### 关键信息

- 左侧是前端入口：
  - 登录
  - 模型配置
  - 数据打点 hook
  - 市场
  - MCP 市场
  - Skill 市场
- 这些前端能力汇入一个 service。
- service 会处理两类核心对象：
  - 用户权限
  - `NoSQL` 数据库
- 用户权限继续细分为：
  - `admin`
  - 部门管理员
  - 用户
- `NoSQL` 数据库会产出配置文件。
- 不同角色对应不同能力：
  - 用户：根据权限获得对应配置
  - 管理员：管理本部门配置
  - `admin`：拥有所有配置权限

### 文本框图

```text
[登录 + 前端配置项]
        |
        v
[服务层]
        |
        +--> [用户权限] -> [admin / 部门管理员 / 用户]
        |
        +--> [NoSQL 数据库] -> [配置文件]
        |
        +--> [用户视角：根据权限提供对应配置]
        +--> [管理员视角：管理本部门配置]
        +--> [admin 视角：所有配置权限]
```

## 最终可落地的方案表达

如果把六张图合并成一句更清晰的产品/技术描述，可以写成：

> 系统提供一个带权限控制的前端配置入口。用户登录并完成 `W3` 鉴权后，系统按部门和用户角色决定其可见配置范围；管理员维护部门白名单、用户权限及本部门配置；前端从本地配置和公共配置中组合出模型、hook、市场、MCP、Skill 等能力，最终由服务层将权限和配置持久化到数据库与配置文件中。
