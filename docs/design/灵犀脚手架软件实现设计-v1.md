## 2. 工程结构与模块划分

### 2.1 实现边界

当前阶段实现边界固定如下：

- 脚手架负责控制面能力：
  - 登录与鉴权
  - 权限解析
  - 候选目录获取与裁剪
  - 默认策略管理
  - 统一配置装配
  - `chrys` 配置适配与启动编排
- `chrys` 负责运行面能力：
  - 消费最终配置
  - 提供运行时动态注入 / provider 能力
  - 提供会话结束回调能力

当前不做：

- 在 `chrys` 内部实现组织、权限、市场等控制面逻辑
- 基于 `AgentProfile.hooks` 的通用 hook 平台
- 任意脚本执行框架

### 2.2 推荐目录结构

建议按“脚手架主工程 + Chrys 对接桥接层”组织代码。

```text
scaffold/
  app/
    tui/
    api/
  domain/
    auth/
    access/
    catalog/
    policy/
    config/
    audit/
  runtime/
    chrys_adapter/
    launch/
    provider_bridge/
    session_callback/
  infra/
    enterprise_api/
    persistence/
    secure_storage/
    cache/
    logging/
  tests/
    unit/
    integration/
    e2e/

chrys_extensions/
  provider/
  session_end/
  tests/
```

目录职责建议：

- `app`：TUI/API 入口与应用编排
- `domain`：领域对象、服务接口、装配规则
- `runtime`：与 `chrys` 直接对接的桥接逻辑
- `infra`：企业 API、数据库、本地安全存储、缓存等基础设施
- `chrys_extensions`：需要在 `chrys` 侧补充的最小扩展实现

### 2.3 核心模块拆分

| 模块 | 主要职责 | 依赖方向 |
|---|---|---|
| `auth` | 统一登录与企业鉴权 | 依赖企业鉴权接口 |
| `access` | 解析 `UserAccessContext` | 依赖组织/权限服务 |
| `catalog` | 获取 `ModelCatalog / CapabilityCatalog` | 依赖能力源与元数据存储 |
| `policy` | 输出 `DefaultPolicy` | 依赖策略存储 |
| `config` | 装配 `ResolvedScaffoldConfig` | 依赖 access/catalog/policy |
| `chrys_adapter` | 生成 `ChrysLaunchBundle` | 依赖 `ResolvedScaffoldConfig` |
| `launch` | 启动前准备与拉起 `chrys` | 依赖 `CredentialBroker` 与 `ChrysRuntimeBridge` |
| `provider_bridge` | 运行时 provider 对接 | 依赖脚手架凭证服务与 `chrys` 动态注入点 |
| `session_callback` | 会话结束回调对接 | 依赖 `chrys` 会话结束事件与审计分发 |
| `audit` | 审计上报 | 依赖审计服务 |

### 2.4 模块依赖约束

建议固定以下依赖方向：

```text
app -> domain -> runtime -> infra
```

补充约束：

- `domain` 不直接依赖 `chrys` 具体实现
- `chrys_adapter / launch / provider_bridge / session_callback` 才允许理解 `chrys` 侧接口
- 企业 API、数据库、本地安全存储统一放在 `infra`

## 3. 核心对象与接口设计

### 3.1 核心数据结构

当前实现建议先固定以下核心对象：

| 对象 | 作用 | 构造方 | 消费方 |
|---|---|---|---|
| `LoginContext` | 登录完成后的最小上下文 | `auth` | `access` |
| `UserAccessContext` | 已解析的访问上下文 | `access` | `catalog / policy / config` |
| `ModelOption` | 单个模型候选项 | `catalog` | `config` |
| `CapabilityItem` | 单个 `skill / mcp` 能力项 | `catalog` | `config` |
| `DefaultPolicy` | 默认装配规则 | `policy` | `config` |
| `UserSelection` | 用户个性化选择 | `app` | `config` |
| `ResolvedScaffoldConfig` | 脚手架统一真相 | `config` | `chrys_adapter / launch` |
| `ChrysLaunchBundle` | 交付给 `chrys` 的适配结果 | `chrys_adapter` | `launch` |
| `CredentialRequirement` | 运行时凭证依赖 | `config` | `launch / provider_bridge` |
| `SessionEndEvent` | 会话结束回调事件 | `chrys_extensions` | `session_callback / audit` |

### 3.2 关键对象关系

当前建议用下面这条主关系组织代码：

```text
LoginContext
  -> UserAccessContext
  -> ModelCatalog / CapabilityCatalog
  -> DefaultPolicy + UserSelection
  -> ResolvedScaffoldConfig
  -> ChrysLaunchBundle
```

其中：

- `ResolvedScaffoldConfig` 是脚手架统一真相
- `ChrysLaunchBundle` 是运行时交付物
- 代码实现中不应跳过 `ResolvedScaffoldConfig` 直接拼接 `chrys` 原生配置

### 3.3 关键接口定义

当前建议先固定以下接口层。

#### `AuthProvider`

```python
class AuthProvider(Protocol):
    async def login(self) -> LoginContext: ...
```

#### `AccessContextResolver`

```python
class AccessContextResolver(Protocol):
    async def resolve(self, login_context: LoginContext) -> UserAccessContext: ...
```

#### `CatalogService`

```python
class CatalogService(Protocol):
    async def get_model_catalog(self, access_context: UserAccessContext) -> list[ModelOption]: ...
    async def get_capability_catalog(self, access_context: UserAccessContext) -> list[CapabilityItem]: ...
```

#### `PolicyService`

```python
class PolicyService(Protocol):
    async def get_default_policy(self, access_context: UserAccessContext) -> DefaultPolicy: ...
```

#### `ConfigAssembler`

```python
class ConfigAssembler(Protocol):
    async def assemble(
        self,
        access_context: UserAccessContext,
        model_catalog: list[ModelOption],
        capability_catalog: list[CapabilityItem],
        default_policy: DefaultPolicy,
        user_selection: UserSelection,
    ) -> ResolvedScaffoldConfig: ...
```

#### `ChrysAdapter`

```python
class ChrysAdapter(Protocol):
    async def adapt(self, config: ResolvedScaffoldConfig) -> ChrysLaunchBundle: ...
```

#### `CredentialBroker`

```python
class CredentialBroker(Protocol):
    async def ensure_credential(self, requirement: CredentialRequirement) -> None: ...
    async def get_valid_credential(self, provider_id: str) -> str: ...
    async def refresh_credential(self, provider_id: str) -> str: ...
```

#### `ChrysRuntimeBridge`

```python
class ChrysRuntimeBridge(Protocol):
    async def launch(self, bundle: ChrysLaunchBundle) -> None: ...
    async def publish_session_end(self, event: SessionEndEvent) -> None: ...
```

#### `AuditDispatcher`

```python
class AuditDispatcher(Protocol):
    async def dispatch(self, event: SessionEndEvent) -> None: ...
```

### 3.4 关键类关系建议

建议代码关系按下面这条线组织：

```text
AppService
  -> AuthService
  -> AccessService
  -> CatalogService
  -> PolicyService
  -> ConfigAssembler
  -> ChrysAdapter
  -> LaunchOrchestrator

LaunchOrchestrator
  -> CredentialBroker
  -> ChrysRuntimeBridge

ChrysRuntimeBridge
  -> DynamicProviderClient
  -> SessionEndCallbackClient

SessionEndCallbackClient
  -> AuditDispatcher
```

补充约束：

- `LaunchOrchestrator` 只编排，不负责业务装配
- `CredentialBroker` 只负责凭证获取、缓存、刷新
- `AuditDispatcher` 只负责上报，不反向影响主会话成功与否

## 4. 关键流程设计

### 4.1 默认配置装配与启动流程

```text
用户登录
-> 获取 LoginContext
-> 解析 UserAccessContext
-> 获取 ModelCatalog / CapabilityCatalog
-> 获取 DefaultPolicy
-> 合并 UserSelection
-> 生成 ResolvedScaffoldConfig
-> 适配为 ChrysLaunchBundle
-> 启动前 ensure_credential
-> 启动 chrys
```

实现要求：

- 候选集合必须先完成权限裁剪
- 默认值必须从已授权候选集合中产生
- 用户选择只能覆盖可选、可覆盖槽位
- 企业固定约束在装配后段统一注入

### 4.2 运行中凭证续期流程

```text
chrys 运行中发起 MCP HTTP 请求
-> 动态 header provider 请求 provider_id 对应凭证
-> Provider Client 调用 CredentialBroker.get_valid_credential
-> 如 token 有效则直接返回
-> 如 token 即将过期或已过期则调用 refresh_credential
-> 生成 headers 并继续请求
-> 如 refresh 失败则返回鉴权错误
```

实现要求：

- 续期逻辑发生在 provider 层，不发生在通用 hook 层
- token 不直接写死在 MCP 静态配置里
- 如遇 401，可触发一次受控 refresh + retry

### 4.3 会话结束审计上报流程

```text
chrys 会话结束
-> Session-End Emitter 生成 SessionEndEvent
-> SessionEndCallbackClient 通知脚手架侧
-> AuditDispatcher 组装审计报文
-> 上报企业审计服务
-> 记录结果
```

实现要求：

- 对 `chrys` 的显式生命周期依赖只落在会话结束回调
- 审计上报默认不阻塞用户退出
- 失败时优先记录日志与重试，不反向把主会话标成失败

### 4.4 状态流转只在流程内定义

当前阶段不单独再起“状态与生命周期设计”章节，统一并入关键流程。

当前只需要在流程实现中显式管理三类状态：

- 凭证状态：
  - `missing / valid / expiring_soon / refreshing / expired / refresh_failed`
- 启动任务状态：
  - `pending / preparing / adapting / launching / running / failed`
- 会话结束上报状态：
  - `pending / dispatching / succeeded / failed`

## 5. 存储与实现约束

### 5.1 数据存储分层

当前建议按四类存储分层：

1. 业务数据库
2. 本地运行文件
3. 本地安全存储
4. 进程内缓存

### 5.2 业务数据库建议承载的对象

当前阶段建议业务数据库只承载控制面数据：

- `CapabilitySource`
- `CapabilityItem`
- `CapabilityPolicy`
- `ModelOption`
- `ModelPolicy`
- `DefaultPolicy`
- `UserSelection`
- 审计任务记录或审计结果索引

补充约束：

- token 明文不进入业务数据库
- `ResolvedScaffoldConfig` 不作为长期事实源入库，必要时只做短期缓存或审计快照

### 5.3 本地运行文件建议承载的对象

本地运行文件主要承载：

- `model_profile_yaml`
- `agent_profile_yaml`
- `.env updates`
- 启动参数或临时运行目录内容

这些对象由 `ChrysLaunchBundle` 派生，不应反向成为控制面事实源。

### 5.4 本地安全存储建议承载的对象

本地安全存储只承载敏感凭证：

- refresh token
- 长期 session credential
- 其他需要跨会话复用的敏感凭证

建议实现方式：

- macOS Keychain
- Windows Credential Manager
- Linux Secret Service / 等价方案

### 5.5 进程内缓存建议承载的对象

进程内缓存主要承载短期状态：

- access token
- provider 查询结果
- 当前启动任务状态
- 当前会话上下文

### 5.6 异常、重试与降级约束

当前建议固定以下处理原则：

- 启动前凭证准备失败：
  - 直接阻塞启动
  - 返回显式错误
- 运行中凭证 refresh 失败：
  - 当前请求失败
  - 不自动扩展成通用全局 hook
- 会话结束上报失败：
  - 不阻塞退出
  - 记录日志并允许异步重试
- `chrys` 启动失败：
  - 启动任务进入 `failed`
  - 保留启动日志与适配结果用于排查

### 5.7 数据库设计放在本章

当前阶段数据库设计建议先放在本章统一收口，不单独拆库表文档。

后续如果出现以下情况，再单独拆《数据库设计》：

- 已明确关系型 / NoSQL / 本地存储的分工
- 已有稳定表/集合清单
- 已需要讨论索引、TTL、迁移与归档

### 5.8 最小测试与验收要求

当前阶段建议至少覆盖：

- `ConfigAssembler` 单元测试
- `ChrysAdapter` 单元测试
- `CredentialBroker` 刷新与失效测试
- 启动前凭证准备联调测试
- 运行中 MCP 动态 header provider 联调测试
- 会话结束回调与审计上报联调测试

## 6. 下一步建议

有了这版软件实现设计，后续最自然的顺序是：

1. 细化 `CredentialBroker / Dynamic Provider` 接口草案
2. 细化 `SessionEndEvent / Session-End Callback` 事件草案
3. 再根据实现需要补充数据库设计或本地运行目录细节
